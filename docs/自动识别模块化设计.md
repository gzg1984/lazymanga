# 自动识别模块化设计

> 更新时间：2026-04-22  
> 目标：将“匹配不了名字时的自动识别”从当前目录规范化链路中的隐式补救逻辑，重构为一个输入输出明确、边界清晰、可独立增强的文本分析模块。

---

## 1. 背景与问题

当前系统已经有两类能力：

- `rulebook` / `filename recognizer`：适合处理结构明确、模式稳定的名字。
- `metadata inference`：在规则没有命中的情况下，尝试根据仓库已有 metadata 与路径内容做补救推断。

当前补救推断链路的主要问题：

1. **模块边界不清晰**
   - 它不是一个独立的“分析器”，而是直接嵌在目录重命名流程里。
   - 输入不是通用协议，而是“repo.db 中已有 metadata 样本 + 相对路径”。

2. **与业务内部结构耦合偏重**
   - 直接依赖 `RepoISO.metadata_json` 的组织方式。
   - 输出结果会立刻参与目录重命名渲染，而不是先形成稳定的分析结果对象。

3. **可增强性不足**
   - 当前算法虽然已经有括号修复、标题残留推断、噪声清理、系列名质量守卫等能力，但这些逻辑散落在实现细节中，不利于后续替换、扩展、测试。

4. **输入源过窄**
   - 当前主要依赖仓库已有样本做推断。
   - 后续如果想引入人工注册词典、规则书提示、人工反馈回流、repo 级热更新提示等能力，现有形态扩展成本较高。

因此，需要把“匹配失败时的自动识别”升级为一个正式模块，而不是继续作为某条规范化流程中的补丁逻辑存在。

---

## 2. 设计目标

本次设计希望实现如下目标：

1. **模块独立**
   - 模块只负责文本分析与信息提取。
   - 不直接依赖 GORM、`RepoISO`、sidecar、文件移动、目录改名等业务执行逻辑。

2. **输入输出明确**
   - 输入由两部分组成：
     - 已注册的分析提示。
     - 待分析的字符串。
   - 输出至少包含：
     - 提取出的 `key/value` 对。
     - 去除已识别信息后的剩余字符串。

3. **便于内部增强**
   - 后续可在不改上层业务协议的前提下，持续增强切词、括号修复、模糊匹配、上下文推断、冲突消解、评分策略。

4. **便于复用**
   - 可用于目录名分析。
   - 也可用于文件名、路径片段、手动输入文本、未来的批量分析场景。

5. **便于维护**
   - 提示注册、分析执行、结果消费三部分分层。
   - 上层业务只关心“给分析器什么提示”和“如何消费分析结果”。

---

## 3. 非目标

本模块本身不负责以下事项：

1. 不直接决定最终目录名或目标路径。
2. 不直接更新数据库。
3. 不直接写 sidecar JSON。
4. 不直接绑定某种 repo 类型。
5. 不替代现有 `filename recognizer` 的强结构匹配能力。

换言之，它是一个“补充型提取引擎”，不是规范化执行器，也不是规则书本身。

---

## 4. 总体架构

建议拆为三层：

### 4.1 提示提供层（Hint Provider）

负责把各种外部知识源整理成统一提示集，例如：

- 仓库中已有的作者、社团、汉化组、原作、Comic Market 信息。
- rulebook 中声明的可识别词表。
- 手工维护的 repo 级词典。
- 未来的用户纠错回流结果。

这一层只负责“生产提示”，不负责文本分析。

### 4.2 分析核心层（Analyzer Core）

负责：

- 文本预处理。
- 括号修复。
- 分段与切词。
- 根据提示匹配可能的值。
- 冲突消解。
- 生成结构化结果。
- 重建剩余文本。

这一层是本设计的核心模块。

### 4.3 业务适配层（Integration Adapter）

负责把分析结果映射回具体业务流程，例如：

- 将剩余文本作为 `title` 候选。
- 将识别出的 `author_name` / `scanlator_group` / `comic_market` 写入 metadata。
- 与现有 `filename recognizer` / `rulebook transform` 联合使用。

这一层负责“怎么用分析结果”，而不是“怎么分析”。

---

## 5. 模块边界

### 5.1 输入

输入应明确拆成两部分：

1. **分析提示注册表（Registered Hints）**
2. **分析请求（Analyze Request）**

### 5.2 输出

输出应为一个稳定的分析结果对象，而不是直接返回某种业务字段集合。

### 5.3 禁止耦合项

分析模块内部不应直接出现：

- `RepoISO`
- `gorm.DB`
- sidecar 文件路径
- 仓库扫描逻辑
- 文件移动逻辑
- rulebook 执行逻辑

这些能力只能出现在提示提供层或业务适配层。

## 6. 输入模型设计

### 6.1 注册提示的抽象

用户当前设想是：

> 某个 key 可能对应某些潜在 value。

这个方向是对的，但建议正式设计时再补齐一些属性，否则后续会缺少冲突处理能力。

建议把提示抽象为如下结构：

```go
type AnalysisHintRegistry struct {
    Fields []AnalysisFieldHint
}

type AnalysisFieldHint struct {
    Key            string
    Label          string
    MultiValue     bool
    Priority       int
    ExclusiveWith  []string
    Values         []AnalysisValueHint
}

type AnalysisValueHint struct {
    CanonicalValue string
    Aliases        []string
    Weight         int
    Source         string
}
```

字段说明：

- `Key`：字段名，例如 `author_name`、`scanlator_group`、`comic_market`。
- `MultiValue`：是否允许一个字符串中提取多个值。
- `Priority`：冲突时字段优先级。
- `ExclusiveWith`：互斥字段，用于控制一个片段不能同时归属多个字段。
- `CanonicalValue`：最终输出的规范值。
- `Aliases`：可匹配的别名、缩写、变体。
- `Weight`：同字段下不同值的优先级，用于处理别名冲突。
- `Source`：提示来源，便于调试和审计。

### 6.2 分析请求

```go
type AnalyzeTextRequest struct {
    Input                string
    AllowFuzzyMatch      bool
    AutoRepairBrackets   bool
    SplitByPathSegments  bool
    PreferLongestMatch   bool
    MaxResults           int
}
```

字段说明：

- `Input`：待分析字符串。
- `AllowFuzzyMatch`：是否允许模糊匹配。
- `AutoRepairBrackets`：是否自动修复明显破损的括号结构。
- `SplitByPathSegments`：是否按路径段参与分析。
- `PreferLongestMatch`：冲突时优先保留更长的候选。
- `MaxResults`：限制返回候选数量，避免调试输出过大。

### 6.3 最小可用版本输入

如果第一阶段希望控制复杂度，可以先实现简化版：

```go
type SimpleHintRegistry map[string][]string
```

但内部最好仍尽量按正式结构组织，避免后续再做一次破坏性升级。

## 7. 输出模型设计

建议输出不仅返回 `key/value` 和剩余字符串，还要包含命中细节与冲突信息，方便上层调试与后续增强。

```go
type AnalyzeTextResult struct {
    Input           string
    NormalizedInput string
    ResidualText    string
    Fields          map[string][]string
    Matches         []AnalyzeMatch
    Rejected        []AnalyzeRejectedMatch
    Warnings        []string
}

type AnalyzeMatch struct {
    Key            string
    Value          string
    Alias          string
    Source         string
    Confidence     float64
    Start          int
    End            int
    MatchedText    string
}

type AnalyzeRejectedMatch struct {
    Key         string
    Value       string
    Reason      string
    MatchedText string
}
```

设计说明：

- `ResidualText`：去掉已识别信息后的剩余字符串，可被上层当作标题候选。
- `Fields`：最终归并后的 key/value 输出。
- `Matches`：保留下来的命中明细。
- `Rejected`：被冲突消解淘汰的候选，便于调试。
- `Warnings`：例如“括号结构已自动修复”“存在重叠匹配”“剩余文本过短”。

## 8. 核心处理流程

建议分析流程固定为以下阶段：

### 8.1 预处理

处理内容：

- 去除首尾空白。
- 统一中西文括号与路径分隔符的处理方式。
- 可选执行括号修复。
- 统一全角/半角空格。

这一阶段应只做“标准化”，不做业务字段判断。

### 8.2 分段

将输入拆为多个候选片段，例如：

- 路径段。
- 括号包裹段。
- 非括号裸文本段。
- 特定分隔符切开的子段。

建议保留每个片段的原始位置，用于后续重建剩余文本。

### 8.3 候选匹配

对每个片段尝试命中注册提示：

- 精确匹配。
- 别名匹配。
- 子串匹配。
- 可选模糊匹配。

输出一组候选命中项，但此时不直接写入最终结果。

### 8.4 冲突消解

需要解决以下典型冲突：

1. 同一片段匹配到多个字段。
2. 同一片段匹配到同字段多个值。
3. 短词与长词重叠。
4. 路径父段和叶子段给出相互冲突的值。

建议优先级顺序如下：

1. 完整匹配优先于部分匹配。
2. 长匹配优先于短匹配。
3. 高权重值优先于低权重值。
4. 高优先级字段优先于低优先级字段。
5. 当前路径叶子段命中优先于更远父段命中。

### 8.5 结果归并

将保留候选归并为最终字段输出：

- 单值字段取最佳值。
- 多值字段保留多个值。
- 如存在字段别名，则统一归并为 canonical key。

### 8.6 剩余文本重建

把已命中的片段从输入中去除，并进行轻量清理：

- 合并多余空格。
- 去除残留空括号。
- 去除仅剩分隔符的空片段。

最终得到 `ResidualText`。

### 8.7 质量守卫

为了避免“识别出了错误字段，反而把标题抽空”，需要有若干守卫规则，例如：

- 剩余文本过短时给出 warning。
- 系列名与标题明显无关时不直接采纳。
- 已知固定噪声标签从 residual 或 title 候选中剔除。
- 低置信度模糊匹配默认不进入最终字段。

## 9. 标题候选策略

你当前的目标之一，是从识别结果里得出“新标题”。

建议不要让分析模块直接输出最终 `title`，而是输出：

1. `ResidualText`
2. 可选的 `TitleCandidate`

例如：

```go
type AnalyzeTextResult struct {
    ...
    ResidualText   string
    TitleCandidate string
}
```

推荐策略：

- 默认 `TitleCandidate = ResidualText`。
- 若 `ResidualText` 为空，则不强行输出标题。
- 标题最终是否采纳，由业务适配层决定。

这样可以避免分析模块过早承担“命名决策”的职责。

## 10. 典型使用方式

### 10.1 漫画仓库自动补救识别

业务层先注册如下提示：

- 已有作者列表。
- 已有 Comic Market 列表。
- 已有汉化组列表。
- 已有社团列表。
- 已有原作列表。

再传入待处理字符串：

```text
【CE家族社】(C86) [牛乳屋さん (牛乳のみお)] J's 4 (女子小学生はじめました)
```

预期输出类似：

```json
{
  "fields": {
    "scanlator_group": ["CE家族社"],
    "comic_market": ["C86"],
    "author_alias": ["牛乳屋さん"],
    "author_name": ["牛乳のみお"],
    "original_work": ["女子小学生はじめました"]
  },
  "residual_text": "J's 4",
  "title_candidate": "J's 4"
}
```

### 10.2 业务层消费方式

业务层可按如下方式使用：

1. 先跑 `filename recognizer`。
2. 如果未命中，再调用自动识别模块。
3. 用分析结果补齐 metadata。
4. 将 `title_candidate` 作为重命名模板的 `title` 候选。
5. 由目录 transform 继续决定最终路径。

这样自动识别模块始终是“提供结构化补充信息”，而不是直接接管整个 rename 流程。

## 11. 与现有能力的关系

### 11.1 与 `filename recognizer` 的关系

二者不是替代关系，而是分工关系：

- `filename recognizer`：强结构、强规则、确定性高。
- 自动识别模块：弱结构、补救型、依赖提示和冲突消解。

建议保留如下执行顺序：

1. 先尝试 `filename recognizer`。
2. 未命中或命中信息不足时，再调用自动识别模块。

### 11.2 与当前 `metadata inference` 的关系

当前 `metadata inference` 里的这些能力可以迁移到新模块内部：

- 括号修复。
- 片段切分。
- 标题残留选择。
- 固定噪声清理。
- 系列名质量守卫。

但“从 repo.db 样本构建模型”的那部分，不应继续充当分析器本体，而应转为一个 Hint Provider：

- 从已有 metadata 样本中抽取字段值。
- 注册为分析提示。
- 再交给统一分析器执行。

### 11.3 与 rulebook 的关系

rulebook 不直接承担自动识别逻辑。

rulebook 可以做的是：

- 决定哪些场景启用自动识别。
- 决定启用哪些 hint provider。
- 决定分析结果如何映射到 transform 模板。

建议先在工程内拆出独立 package，例如：

```text
backend/normalization/textanalyzer/

  analyzer.go
  preprocess.go
  segment.go
  matcher.go
  resolver.go
  residual.go
  quality.go
  provider_repo_metadata.go
  provider_static_dict.go
  analyzer_test.go
  provider_repo_metadata_test.go
```

职责建议：

- `types.go`：公开输入输出协议。
- `analyzer.go`：主入口。
- `preprocess.go`：括号修复与标准化。
- `segment.go`：文本切段。
- `matcher.go`：提示匹配。
- `resolver.go`：冲突消解。
- `residual.go`：剩余文本重建。
- `quality.go`：质量守卫。
- `provider_*`：各种提示来源适配器。

## 13. 接口建议

### 13.1 核心分析接口

```go
type Analyzer interface {
    Analyze(req AnalyzeTextRequest, registry AnalysisHintRegistry) (AnalyzeTextResult, error)
}
```

### 13.2 提示提供接口

```go
type HintProvider interface {
    BuildHints() (AnalysisHintRegistry, error)
}
```

如需带上下文，可扩展为：

```go
type HintProvider interface {
    BuildHints(ctx HintBuildContext) (AnalysisHintRegistry, error)
}

type HintBuildContext struct {
    RepoID uint
    Root   string
}
```

### 13.3 多提示源合并器

```go
type RegistryMerger interface {
    Merge(registries ...AnalysisHintRegistry) AnalysisHintRegistry
}
```

这有利于把“仓库样本提示”“静态字典提示”“规则书附加提示”组合在一起。

## 14. 第一阶段推荐实现范围

为了避免一次性过大，建议 v1 只做以下范围：

1. 提示注册表支持：
   - 单值字段。
   - alias 匹配。
   - 权重与优先级。

2. 分析能力支持：
   - 文本标准化。
   - 括号修复。
   - 按括号与路径段切分。
   - 精确匹配与子串匹配。
   - 冲突消解。
   - 剩余文本重建。

3. provider 支持：
   - 从 repo 既有 metadata 构建 hints。
   - 从静态字典构建 hints。

4. 业务集成支持：
   - 在 `filename recognizer` 未命中时作为 fallback。
   - 输出只用于补齐 metadata 与生成 title 候选。

v1 明确不做：

- 高成本模糊匹配。
- 机器学习模型。
- 自动在线学习。
- 多语言形态学分析。

## 15. 迁移方案

建议分三步迁移：

### Phase 1：抽离协议与核心处理

- 新建 `textanalyzer` 包。
- 将当前 `metadata inference` 中通用能力迁入：
  - 文本清洗。
  - 括号修复。
  - 分段。
  - 标题残留策略。
  - 固定噪声守卫。

此阶段目标：先让分析器脱离目录 transform 代码。

### Phase 2：将 repo metadata 学习改造成 Hint Provider

- 现有从 `RepoISO.metadata_json` 聚合字段值的逻辑，不再直接推断结果。
- 改为向分析器提供字段值候选。

此阶段目标：把“样本知识来源”和“分析过程”彻底分离。

### Phase 3：业务接线替换

- `applyDirectoryTransformWithAnalysis` 不再直接调用当前推断器。
- 改为：
  1. 收集 hints。
  2. 调用 `Analyzer.Analyze(...)`。
  3. 将结果映射到 metadata。
  4. 再交给目录 transform 渲染。

此阶段完成后，旧 `metadata inference` 可逐步瘦身或废弃。

## 16. 测试策略

至少覆盖以下测试：

1. **预处理测试**
   - 破损括号可修复。
   - 空白与分隔符标准化稳定。

2. **匹配测试**
   - 精确值匹配。
   - alias 匹配。
   - 子串匹配。

3. **冲突测试**
   - 长词覆盖短词。
   - 同片段多字段冲突。
   - 同字段多值冲突。

4. **剩余文本测试**
   - 已识别片段被正确剔除。
   - 残余空括号和多余空格被清理。

5. **业务样例测试**
   - 漫画目录名可提取出汉化组、作者、Comic Market、原作。
   - 剩余文本可稳定得到标题候选。

6. **回归测试**
   - 当前已通过的 `karita` 相关识别样例不应被破坏。

## 17. 风险与注意事项

### 17.1 不要过早把“分析结果”直接等同于“最终 metadata”

分析器输出的是候选结构化信息，不是最终业务事实。

业务层仍应决定：

- 是否采纳某字段。
- 是否把 residual 当 title。
- 是否覆盖已有 metadata。

### 17.2 不要让 provider 重新侵入分析器内部

一旦分析器内部直接知道“repo metadata 样本长什么样”，边界又会退化回当前状态。

### 17.3 不要把强规则识别和补救识别混成一套 if/else 逻辑

推荐始终维持：

- 强规则识别优先。
- 自动识别作为 fallback 或信息补充。

## 18. 本设计的结论
