# 文本分析字段策略统一入口设计

> 更新时间：2026-04-23  
> 目标：在不重写当前文本分析链路的前提下，为字段语义提供一个小规模、低干扰、可渐进扩展的统一入口，降低后续增加新字段语义和局部重构的成本。

---

## 1. 背景

当前文本分析能力主要分布在三块：

- `metadata inference`：基于 repo 内已有 metadata 样本和路径文本，做弱结构推断。
- `filename recognizer`：基于模式规则或规则书，做强结构提取。
- refresh proposal / manual edit adapter：把分析结果转成候选提案并进入确认流。

现状已经不是“所有 key/value 完全平等”的纯通用系统，而是一个：

- 底层采用通用统计索引；
- 上层存在若干字段特化逻辑；
- 业务消费层再做 proposal/编辑语义转换。

这种形态本身是合理的，但当前“字段为什么特殊”这一层是散落的：

1. 有些字段不会进入基础分析模型。
2. 有些字段会被当作 context anchor。
3. 有些字段会参与 title 候选打分。
4. 有些字段会被当作技术字段，仅用于追踪，不用于语义推断。
5. 有些字段未来可能允许“仅作为候选覆盖提案”而不是“仅补空”。

这些语义当前分散在多个函数中，维护时需要来回跳转，新增字段语义时也容易继续引入新的散点判断。

因此，需要一个“小规模整理 + 统一入口”的改造，而不是立刻重写整个分析框架。

---

## 2. 本次方案的核心判断

本次不做“大一统推断框架”，只做一层很薄的字段策略入口层。

### 2.1 这层的目标

只解决两个问题：

1. **集中表达字段语义**
   - 哪些字段是技术字段。
   - 哪些字段允许进入分析模型。
   - 哪些字段是 context anchor。
   - 哪些字段会参与 title / series_name 关系判断。
   - 哪些字段会影响 refresh proposal 的展示或覆盖策略。

2. **为后续扩展提供稳定挂点**
   - 以后引入新字段或新语义，不再到处找 `if field == "title"` 这类散点，而是先改统一策略入口，再由现有算法消费。

### 2.2 这层明确不做什么

1. 不重写当前 `metadata inference` 流程。
2. 不把所有推断逻辑插件化。
3. 不尝试把 `filename recognizer` 和 `metadata inference` 统一成一套 DSL。
4. 不要求所有未来功能都必须经过一套新框架才能落地。

换句话说，这不是“新架构替换旧架构”，而是“给旧架构补一层统一语义目录”。

---

## 3. 为什么这是小规模且低负担的改造

如果控制边界，这种整理不会成为未来大改的包袱，原因如下：

1. **它不改主流程，只改判断来源**
   - 当前建模、推断、质量守卫、proposal 生成这些主流程仍然保留。
   - 改造点只是把散落判断替换成统一入口查询。

2. **它是“薄入口”，不是“强制框架”**
   - 新功能仍然可以旁路接入，不要求必须重写成插件。
   - 统一入口只是优先的字段语义来源，而不是唯一扩展协议。

3. **它会降低大改成本，而不是抬高成本**
   - 将来如果要做更大的架构演进，至少“字段语义分布图”已经被集中表达。
   - 大改时可以先迁移这一层，再迁移下层算法。

因此，这种改造的工程风险主要来自迁移时的行为一致性，而不是架构侵入性本身。

---

## 4. 统一入口的设计目标

统一入口层建议只提供以下能力：

### 4.1 字段角色定义

最少需要表达这些角色：

- `technical`：技术字段，不参与语义推断主模型。
- `semantic`：语义字段，可以进入统计模型。
- `context anchor`：可作为上下文锚点，例如 `title`、`series_name`。
- `title-related`：可参与 title 候选评分或标题关系判断。
- `proposal-visible`：refresh proposal 中可以展示。
- `proposal-fill-empty-only`：仅允许补空，不直接覆盖已有值。
- `proposal-candidate-overwrite`：未来如果启用，可用于候选覆盖展示。

### 4.2 统一查询入口

而不是在各处重复写：

- “这个字段能不能进入模型？”
- “这个字段是不是技术字段？”
- “这个字段是不是 title anchor？”

建议统一成类似：

```go
type FieldSemanticRole struct {
    Key                    string
    Technical              bool
    IncludeInAnalysisModel bool
    ContextAnchor          bool
    TitleRelated           bool
    ProposalVisible        bool
    FillEmptyOnly          bool
}

func ResolveFieldSemanticRole(key string) FieldSemanticRole
```

第一阶段不要求这个结构非常完整，但至少要能承载当前已经出现的语义差异。

---

## 5. 建议落点

建议新增一个专门文件，放在：

- `backend/normalization/field_semantics.go`

原因：

1. 当前字段语义主要首先影响 `normalization` 层。
2. 这个文件名足够中性，不会把设计绑定死在 metadata inference 上。
3. 后续如果 `filename recognizer`、proposal adapter 也要复用字段语义，可以继续从这里读取，而不是复制一套判断。

---

## 6. 第一阶段只收敛哪些散点判断

第一阶段不追求“全量统一”，只收敛最容易形成维护负担、又最稳定的几类判断。

### 6.1 建模准入判断

当前散点：

- `shouldUseMetadataFieldForAnalysis()`

第一阶段目标：

- 改成从字段语义入口读取“是否进入分析模型”。

### 6.2 技术字段过滤

当前散点：

- refresh diagnostics 中对 `source_path`、`original_name`、`archive_storage_path` 等字段的排除。

第一阶段目标：

- 统一改成根据字段语义入口判断“是否属于 technical / non-semantic”。

### 6.3 context anchor 判定

当前散点：

- `title`
- `series_name`

这些字段在：

- `ContextValueCounts`
- `applyContextMetadataHints()`
- `scoreTitleCandidate()`

中都被特殊对待。

第一阶段目标：

- 不立刻把算法通用化。
- 先把“哪些字段是 anchor”收敛成统一判断来源。

### 6.4 proposal 基础策略

当前散点：

- 某些字段只补空、不覆盖已有值。

第一阶段目标：

- 不改变当前 refresh proposal 行为。
- 先把“哪些字段当前默认只补空”表达成统一字段属性，为以后引入“候选覆盖提案”打基础。

---

## 7. 第一阶段刻意不抽的内容

这些内容短期内不要抽，否则容易过度设计：

### 7.1 不抽 `inferTitleFromLeaf()` 的内部流程

原因：

- 这一层目前仍然强依赖路径分段、括号结构、噪声清理、ignored prefix 等启发式。
- 直接抽象成通用“标题策略插件”会让复杂度显著上升。

### 7.2 不抽 `applyMetadataQualityGuards()` 为通用规则 DSL

原因：

- 当前 `title` / `series_name` 的质量规则仍然是领域规则。
- 第一阶段只要把“这些规则针对哪些字段”集中管理，不需要把规则本身声明式化。

### 7.3 不统一 `filename recognizer` 与 `metadata inference` 的算法协议

原因：

- 两者一个偏模式匹配，一个偏统计+启发式。
- 它们共享“字段语义定义”很合理，但共享“执行协议”在第一阶段不划算。

### 7.4 不把所有字段强制塞进同一角色系统

原因：

- 有些字段未来可能需要更多业务语义，但当前还没有稳定定义。
- 第一阶段应该允许字段语义定义逐步补齐，而不是一次建完一张大表。

---

## 8. 第一阶段建议结构

建议先只实现一张小而明确的字段策略表，例如：

```go
type FieldSemanticRole struct {
    Key                    string
    Technical              bool
    IncludeInAnalysisModel bool
    ContextAnchor          bool
    TitleRelated           bool
    ProposalVisible        bool
    FillEmptyOnly          bool
}

var defaultFieldSemanticRoles = map[string]FieldSemanticRole{
    "title": {
        Key:                    "title",
        IncludeInAnalysisModel: true,
        ContextAnchor:          true,
        TitleRelated:           true,
        ProposalVisible:        true,
        FillEmptyOnly:          true,
    },
    "series_name": {
        Key:                    "series_name",
        IncludeInAnalysisModel: true,
        ContextAnchor:          true,
        TitleRelated:           true,
        ProposalVisible:        true,
        FillEmptyOnly:          true,
    },
    "source_path": {
        Key:             "source_path",
        Technical:       true,
        ProposalVisible: true,
    },
    "original_name": {
        Key:             "original_name",
        Technical:       true,
        ProposalVisible: true,
    },
}
```

注意：这只是第一阶段的表达层，不要求所有属性立即都被所有模块消费。

---

## 9. 建议迁移顺序

建议严格按以下顺序迁移，以降低行为风险：

### 第 1 步：新增字段语义入口文件

目标：

- 只新增定义和查询函数。
- 不改现有行为。

### 第 2 步：替换建模准入判断

目标：

- `shouldUseMetadataFieldForAnalysis()` 内部改为调用统一字段语义入口。

### 第 3 步：替换 diagnostics 技术字段过滤

目标：

- refresh diagnostics 中不再手写排除清单，而是读统一字段语义入口。

### 第 4 步：替换 context anchor 判定来源

目标：

- `title` / `series_name` 的特殊入口判断从散点硬编码收敛到统一入口。
- 但算法主体暂时不动。

### 第 5 步：给 proposal 策略留挂点

目标：

- 先表达 FillEmptyOnly 等属性。
- 暂不改变 refresh proposal 行为。

这样做可以保证每一步都是小改、可回滚、可单独验证。

---

## 10. 风险与控制

### 风险 1：行为悄悄变化

控制方式：

- 第一阶段不改算法主体，只改判断来源。
- 每替换一类判断，都补最小测试。

### 风险 2：入口层反过来绑架业务

控制方式：

- 入口层只定义字段语义，不定义执行框架。
- 保持“可旁路”原则，不强迫所有逻辑都走统一插件协议。

### 风险 3：抽象过宽，字段角色过度设计

控制方式：

- 第一阶段只定义已被代码真实使用的属性。
- 不预设大量未来属性。

---

## 11. 预期收益

如果只做这层小规模整理，预期收益主要有：

1. 新增字段语义时，不再继续扩散散点判断。
2. refresh diagnostics、proposal 策略、metadata inference 可以共享同一份字段语义定义。
3. 后续如果要引入“候选覆盖提案”“字段级置信度”“字段来源审计”，有明确挂点。
4. 将来若要做更大规模重构，这层可以作为稳定过渡面。

---

## 12. 结论

当前最合适的方向，不是立刻重写一套新的文本分析框架，而是先增加一个很薄的字段语义统一入口。

这项改动：

- **规模小**：主要是整理散点判断来源。
- **干涉低**：不直接重写现有算法流程。
- **收益稳定**：能为后续继续增强 title / series / proposal 行为提供清晰挂点。

因此，推荐把它作为文本分析体系的第一阶段整理任务，而不是等到未来更大重构时再一次性处理。
