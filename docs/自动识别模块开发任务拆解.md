# 自动识别模块开发任务拆解

> 更新时间：2026-04-22  
> 对应设计：`docs/自动识别模块化设计.md`  
> 目标：把“匹配失败后的自动识别”从当前目录规范化链路中的隐式补救逻辑，按可执行任务拆成若干阶段，指导后续逐步开发与替换。

---

## 1. 文档用途

本文件不是再讲一遍设计，而是把设计转换成后续可执行开发任务。

重点解决三个问题：

1. 先做什么，后做什么。
2. 每一步改哪些文件。
3. 每一步完成后，如何判断没有跑偏。

本文件默认以后端为主，因为当前自动识别的核心耦合点主要在 `backend/normalization`。

---

## 2. 当前实现锚点

当前“匹配失败后的自动识别”直接挂在目录规范化链路上，主要锚点如下：

1. `DirectoryTransformStep.Process(...)`
   - 位置：`backend/normalization/directory_transform_step.go`
   - 当前会先构建 repo 级分析模型，再调用带分析的 transform。

2. `applyDirectoryTransformWithAnalysis(...)`
   - 位置：`backend/normalization/directory_transform_step.go`
   - 当前逻辑是：
     - 先尝试 `applyDirectoryTransform(...)`。
     - 再调用 `AnalyzePathMetadata(...)` 做补救推断。
     - 把 existing metadata、推断 metadata、recognizer captures 合并。
     - 再渲染最终名称。

3. `BuildRepoPathAnalysisModel(...)` / `AnalyzePathMetadata(...)`
   - 位置：`backend/normalization/metadata_inference.go`
   - 当前同时承担：
     - 从 repo.db 样本聚合“知识”。
     - 对路径做实际分析。

这正是当前边界不清的根源。

---

## 3. 开发总原则

后续开发应遵循以下原则：

1. **先抽协议，再迁逻辑，再替换业务接线**
   - 不要一开始就在 `directory_transform_step.go` 里硬改成新逻辑。

2. **先保行为，再做重命名**
   - 第一阶段优先保证现有样例继续工作。
   - 不要求一开始就把旧实现完全删除。

3. **新模块先做并行替代，不做一次性爆改**
   - 允许 Phase 1 暂时和旧 `metadata_inference` 并存。

4. **强规则识别优先顺序不变**
   - `filename recognizer` 仍应先于自动识别运行。

5. **分析器只产出结构化结果，不直接决定最终 rename**
   - 目录改名仍应由上层 transform 模板渲染决定。

---

## 4. 阶段总览

建议按四个开发阶段推进：

1. **Phase 1A：抽离文本分析协议与核心骨架**
2. **Phase 1B：迁移现有通用文本处理能力**
3. **Phase 1C：把 repo metadata 学习改造成 Hint Provider**
4. **Phase 1D：接入目录 transform fallback 链路**

其中：

- Phase 1A ~ 1C 可以基本不动业务主链路。
- Phase 1D 才是第一次替换真实调用路径。

---

## 5. Phase 1A：抽离文本分析协议与核心骨架

### 5.1 目标

先把“自动识别”定义成一个正式 package，并建立稳定输入输出协议。

这一阶段的目标不是立刻替换旧逻辑，而是先让新模块在工程内存在，并能被单元测试调用。

### 5.2 预期产物

建议新增 package：

```text
backend/normalization/textanalyzer/
  types.go
  analyzer.go
  preprocess.go
  segment.go
  matcher.go
  resolver.go
  residual.go
  quality.go
```

### 5.3 最小接口

第一阶段建议先落以下公开类型：

```go
type AnalysisHintRegistry struct { ... }
type AnalysisFieldHint struct { ... }
type AnalysisValueHint struct { ... }

type AnalyzeTextRequest struct { ... }
type AnalyzeTextResult struct { ... }
type AnalyzeMatch struct { ... }
type AnalyzeRejectedMatch struct { ... }

type Analyzer interface {
    Analyze(req AnalyzeTextRequest, registry AnalysisHintRegistry) (AnalyzeTextResult, error)
}
```

### 5.4 这一阶段建议修改的文件

新增：

- `backend/normalization/textanalyzer/types.go`
- `backend/normalization/textanalyzer/analyzer.go`

新增测试：

- `backend/normalization/textanalyzer/analyzer_test.go`

### 5.5 这一阶段不要做的事

1. 不要接入 `directory_transform_step.go`
2. 不要依赖 `gorm.DB`
3. 不要读取 `RepoISO`
4. 不要先做模糊匹配

### 5.6 完成标准

完成后应满足：

1. `textanalyzer` 包可以独立编译。
2. 至少有一条最小测试可跑通：
   - 输入若干 hint。
   - 输入一条字符串。
   - 返回 `fields + residual_text`。

## 6. Phase 1B：迁移现有通用文本处理能力

### 6.1 目标

把当前 `metadata_inference.go` 中和“文本处理”直接相关、又不依赖 repo 样本建模的能力迁入新模块。

这一步的核心是把“分析器的普适能力”从旧文件里剥出来。

### 6.2 优先迁移的能力

优先迁移下列能力：

1. 文本清洗
2. 括号修复
3. 片段切分
4. 去括号残留文本提取
5. 固定噪声标签清理
6. 系列名/标题质量守卫

### 6.3 直接可复用的旧实现素材

当前这些逻辑大多在：

- `backend/normalization/metadata_inference.go`

其中可优先参考或迁出的能力包括：

- `splitPathInferenceSegments(...)`
- `stripBracketedSegments(...)`
- `bestBareSegment(...)`
- `stripFixedTitleNoiseTags(...)`
- `applyMetadataQualityGuards(...)`
- 各类 `normalize...` / `clean...` 文本工具函数

注意：这一步是“迁移通用文本能力”，不是照抄整个 `metadata_inference.go`。

### 6.4 建议修改的文件

新增/编辑：

- `backend/normalization/textanalyzer/preprocess.go`
- `backend/normalization/textanalyzer/segment.go`
- `backend/normalization/textanalyzer/residual.go`
- `backend/normalization/textanalyzer/quality.go`

新增测试：

- `backend/normalization/textanalyzer/preprocess_test.go`
- `backend/normalization/textanalyzer/segment_test.go`
- `backend/normalization/textanalyzer/quality_test.go`

### 6.5 验收样例

至少覆盖以下样例：

1. 括号破损修复：
   - `[(C86]` 能被修正到可分析状态。

2. 中文漫画名残留：
   - `【组】 [社团] 标题 [(原作]` 能得到正确 residual/title candidate。

3. 噪声标签清理：
   - `[中国翻訳] [DL版] [無修正]` 等可被清理。

### 6.6 完成标准

完成后应满足：

1. 新包已能独立完成基础文本预处理与 residual 重建。
2. 这些能力不再依赖 repo 样本建模。
3. 即使暂时没接业务主链路，也已经能用单元测试覆盖核心行为。

## 7. Phase 1C：把 repo metadata 学习改造成 Hint Provider

### 7.1 目标

把当前“从 repo.db 里已有 metadata 样本学习字段值”的逻辑，从“分析器本体”降级成“提示提供器”。

这是整个重构里最关键的一步，因为它决定边界是否真的被拉开。

### 7.2 当前问题

当前 `metadata_inference.go` 里：

- `BuildRepoPathAnalysisModel(...)` 会从 repo.db 读取样本。
- `AnalyzePathMetadata(...)` 直接消费这个 model 做推断。

这意味着“知识来源”和“分析动作”耦合在一个模型内。

### 7.3 目标改造方向

改造成：

1. provider 从 repo.db 样本中提取字段值、别名、常见写法。
2. provider 输出 `AnalysisHintRegistry`。
3. `Analyzer.Analyze(...)` 只消费 registry 和 input string。

### 7.4 建议新增结构

建议新增：

```go
type HintProvider interface {
    BuildHints(ctx HintBuildContext) (AnalysisHintRegistry, error)
}

type HintBuildContext struct {
    RepoID uint
    RepoDB *gorm.DB
}
```

### 7.5 建议新增文件

新增：

- `backend/normalization/textanalyzer/provider_repo_metadata.go`
- `backend/normalization/textanalyzer/provider_merge.go`

新增测试：

- `backend/normalization/textanalyzer/provider_repo_metadata_test.go`

### 7.6 Phase 1C 只做什么

这一阶段建议只做：

1. 从 `metadata_json` 中抽取字段值。
2. 形成按字段组织的 hint registry。
3. 允许简单 alias 回填策略，例如：
   - `event_code -> comic_market`
   - `circle_name -> circle`

### 7.7 Phase 1C 暂时不做什么

1. 不做复杂 co-occurrence 上下文投票迁移。
2. 不做自动学习回写。
3. 不做 provider 的热更新优化。

### 7.8 完成标准

完成后应满足：

1. 给定一组 `RepoISO.metadata_json` 样本，可以构建出 registry。
2. 分析器不需要知道 `RepoISO` 是什么。
3. repo metadata provider 可以单独测试。

## 8. Phase 1D：接入目录 transform fallback 链路

### 8.1 目标

将 `directory_transform_step.go` 中当前直接调用旧推断器的部分，替换为：

1. 构建 hints。
2. 调用新分析器。
3. 将结果映射为 metadata。
4. 再交给现有目录 transform 渲染。

### 8.2 当前替换点

当前直接替换点就是：

- `backend/normalization/directory_transform_step.go`
- 函数：`applyDirectoryTransformWithAnalysis(...)`

当前伪流程：

1. `applyDirectoryTransform(...)`
2. `AnalyzePathMetadata(model, relativePath)`
3. 合并 `existingMetadata + guess.Metadata + captures`
4. `renderDirectoryTransformFromMetadata(...)`

目标流程：

1. `applyDirectoryTransform(...)`
2. `BuildHints(...)`
3. `Analyzer.Analyze(...)`
4. 把 `AnalyzeTextResult` 映射成 metadata map
5. 合并 `existingMetadata + analyzerFields + captures`
6. `renderDirectoryTransformFromMetadata(...)`

### 8.3 建议新增适配函数

建议新增一个本地适配层函数，例如：

```go
func analyzeDirectoryPathFallback(...) (map[string]string, string, error)
```

职责：

- 不直接暴露 analyzer 细节给目录 transform。
- 把 analyzer 输出转成当前业务能消费的 metadata map。
- 必要时返回 `title_candidate`。

### 8.4 建议修改文件

编辑：

- `backend/normalization/directory_transform_step.go`

新增或编辑测试：

- `backend/normalization/directory_transform_step_test.go`

### 8.5 第一批应保持通过的场景

至少保证以下现有场景不退化：

1. 旧 `karita` 路径样例仍能识别标题与主要 metadata。
2. 当已有 `existingMetadata` 时，不会因为分析器新接入而丢失原 metadata。
3. 当前已整理路径再次 refresh 时，不会因为 residual 为空而把 title 弄丢。

### 8.6 完成标准

完成后应满足：

1. `directory_transform_step.go` 已不再直接依赖旧 `AnalyzePathMetadata(...)` 作为唯一 fallback。
2. 目录 transform 仍保持“强规则优先，自动识别补充”的结构。
3. 现有关键目录识别测试继续通过。

## 9. 旧实现处理策略

### 9.1 短期策略

短期内不建议立刻删除旧 `metadata_inference.go`。

建议策略：

1. 先保留。
2. 按阶段把通用能力迁出。
3. 等新 fallback 路径稳定后，再判断是否瘦身或废弃旧入口。

### 9.2 可接受的过渡形态

在过渡期内允许出现以下状态：

- `metadata_inference.go` 还在。
- `textanalyzer/` 已存在。
- 目录 transform 部分逻辑已改走新模块。
- 少量工具函数暂时仍由旧文件复用。

这比一次性硬删旧实现更稳。

---

## 10. 推荐开发顺序

推荐按以下顺序提交：

1. 新建 `textanalyzer` 包与基础类型。
2. 迁移文本预处理、分段、质量守卫。
3. 做静态 hint 的最小 analyzer 测试。
4. 增加 repo metadata provider。
5. 在目录 transform 中接入新 analyzer。
6. 补齐回归测试。

不建议的顺序：

1. 先改 `directory_transform_step.go`
2. 再临时发明新结构
3. 最后才去补 analyzer 协议

这种顺序很容易把新模块又写回业务代码内部。

## 12. 风险点清单

### 12.1 最大风险

最大风险不是“算法不够聪明”，而是边界又退化回去。

典型表现：

1. analyzer 内部再次直接读取 repo.db。
2. directory transform 里再次塞入大量 analyzer 内部细节。
3. provider 和 analyzer 共用一个巨大 struct。

### 12.2 第二风险

第二风险是 residual/title 处理变化导致现有标题回归。

因此接入业务主链路前，必须保住：

- 当前 `karita` 目录标题样例
- 当前 broken bracket 样例
- 当前 existing metadata 保留样例

### 12.3 第三风险

第三风险是过早追求复杂匹配能力，导致 Phase 1 范围膨胀。

Phase 1 必须坚持：

- 不做高成本模糊匹配
- 不做自动学习回写
- 不做多语言复杂 NLP

## 13. 建议的任务单结构

如果后续要进一步拆成 issue 或 checklist，建议按下面粒度建任务：

1. `textanalyzer: add public types and analyzer skeleton`
2. `textanalyzer: migrate bracket repair and text normalization helpers`
3. `textanalyzer: add segment splitting and residual reconstruction`
4. `textanalyzer: add quality guards and title candidate logic`
5. `textanalyzer: add repo metadata hint provider`
6. `normalization: wire textanalyzer into directory transform fallback`
7. `tests: add regression coverage for auto-recognition migration`

这个粒度适合逐步提交，也适合回退。

---

## 14. 本文档结论

后续开发不应直接围绕旧 `metadata_inference.go` 补丁式扩展，而应沿着下面这条路径推进：

1. 先把自动识别正式定义成 `textanalyzer` 模块。
2. 再把当前可复用文本能力迁入新模块。
3. 再把 repo metadata 样本学习降级成 Hint Provider。
4. 最后再替换 `applyDirectoryTransformWithAnalysis(...)` 的 fallback 接线。

如果严格按这个顺序推进，就能在保住现有行为的前提下，把当前补丁式自动识别逐步升级为真正独立、可维护、可增强的模块。
