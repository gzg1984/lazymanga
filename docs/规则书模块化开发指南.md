# 规则书模块化开发指南

本文档用于指导 lazymanga 的“文件自动移动”能力从当前业务代码中抽离为独立模块，并为后续发布为工程外 Go module 做准备。

目标：
- 将“规则匹配”和“移动决策”从业务流程中解耦。
- 支持每个仓库绑定一本规则书。
- 支持预置规则书与临时覆盖规则（运行时修改）。
- 保持现有行为可兼容，允许渐进式迁移。

---

## 1. 当前现状与问题

当前自动移动逻辑主要集中在：
- `backend/normalization/os_relocation_step.go`
- `backend/normalization/pipeline.go`
- `backend/handlers/repo_normalize.go`

当前形态的优点：
- 已有清晰的 normalize pipeline。
- 已有文件移动、冲突后缀、跨设备移动 fallback（EXDEV）。

当前形态的问题：
- 规则（关键词 -> 目标目录）硬编码在 step 内部。
- 决策逻辑与数据库更新、文件操作耦合。
- 不方便按仓库使用不同规则集。
- 不方便临时变更规则并回滚。

---

## 2. 目标架构

采用分层设计：

1. 规则引擎层（可独立模块）
- 只做规则匹配和决策输出。
- 不依赖 GORM，不依赖具体 RepoISO 模型。
- 不直接操作文件系统。

2. 执行适配层（留在业务工程）
- 根据引擎输出执行移动。
- 写回数据库字段（path/file_name/is_os 等）。
- 处理日志、事务、异步任务。

3. 规则书管理层（可先在工程内）
- 加载预置规则书。
- 提供仓库级规则书绑定。
- 支持临时覆盖规则（overlay）。

---

## 3. 目录建议（第一阶段：工程内）

建议在 `backend/normalization` 下先拆出：

```
backend/normalization/rulebook/
  types.go              # RuleBook / Rule / Condition / Action / Result
  engine.go             # 匹配与决策
  matcher.go            # 字段匹配器（文件名关键词、路径等）
  default_books.go      # 当前默认规则书（迁移自硬编码）
  overlay.go            # 临时覆盖规则 merge
  errors.go             # 可分类错误
```

第二阶段再迁出为独立仓库（例如 `github.com/xxx/lazymanga-rulebook`）。

---

## 4. 核心数据模型（建议）

```go
package rulebook

type RuleBook struct {
    Name    string
    Version string
    Rules   []Rule
}

type Rule struct {
    ID       string
    Priority int
    Enabled  bool
    Match    Condition
    Action   Action
}

type Condition struct {
    IsOS          *bool
    IsEntertainment *bool
    FileNameContains []string
    PathContains     []string
}

type Action struct {
    TargetDir        string
    InferIsOS        bool
    InferEntertainment bool
    Reason           string
}

type EvalInput struct {
    RelativePath   string
    FileName       string
    IsOS           bool
    IsEntertainment bool
}

type EvalResult struct {
    Matched     bool
    RuleID      string
    RuleBook    string
    RuleVersion string
    Action      Action
}
```

设计要点：
- `Priority` 数值越小优先级越高（推荐统一约定）。
- `Enabled=false` 时忽略规则。
- 引擎输出必须包含命中规则和版本信息，方便审计与回放。

---

## 5. 引擎接口（建议）

```go
type Engine interface {
    Evaluate(in EvalInput) (EvalResult, error)
}
```

可选增强接口：

```go
type RuleBookProvider interface {
    Get(bookName string) (RuleBook, error)
    GetForRepo(repoID uint) (RuleBook, error)
}

type OverlayProvider interface {
    GetOverlay(repoID uint) ([]Rule, error)
}
```

## 6. 与现有流程的集成方式

将 `OSRelocationStep` 改为三段：

1. 构建 `EvalInput`
2. 调用 rulebook engine 得到 `EvalResult`
3. 若命中则执行移动并更新数据库

伪代码：

```go
result, err := engine.Evaluate(EvalInput{...})
if err != nil { return err }
if !result.Matched { return nil }

targetDir := result.Action.TargetDir
finalAbs, finalRel, moved, err := relocate(...)
if err != nil { return err }

updates := map[string]interface{}{
  "path": finalRel,
  "file_name": filepath.Base(finalRel),
}
if result.Action.InferIsOS { updates["is_os"] = true }
if result.Action.InferEntertainment { updates["is_entertament"] = true }
```

日志建议强制带上：
- `rule_book`
- `rule_version`
- `rule_id`
- `reason`

## 7. 规则书生命周期与“临时修改”

支持两种来源：

1. 预置规则书（持久化）
- 存数据库或 JSON 文件。
- 例如：`default-os-v1`。

2. 临时覆盖规则（overlay）
- 仅对指定 repo 生效。
- 可以设置 TTL（例如 30 分钟）或 `expires_at`。
- 到期自动失效，回退到基础规则书。

冲突处理建议：
- 先按 `Priority` 排序。
- 同优先级按声明顺序。
- overlay 规则整体优先于基础规则书。

---

## 8. 并发与一致性建议

1. 运行快照
- pipeline 启动时固定本批次使用的规则书版本。
- 不要在批处理中途切换版本。

2. 原子替换
- 规则书可通过 `atomic.Value` 或 `RWMutex` 做热更新。

3. 幂等
- 相同输入多次执行，最终路径一致。
- 避免反复改名产生 `_1`, `_2` 连锁膨胀。

---

## 9. 从工程内模块到工程外 Go module 的迁移路径

### 阶段 A（现在）：仓库内抽象
- 目标：把规则匹配从 `os_relocation_step` 抽离。
- 产出：`rulebook` package + 单元测试。

### 阶段 B：接口稳定
- 目标：冻结 `EvalInput/EvalResult` 等公共类型。
- 产出：版本策略（v1）。

### 阶段 C：迁出独立仓库
- 目标：发布独立 Go module。
- 产出：
  - `go.mod`
  - API 文档
  - 示例（如何在 lazymanga 中接入）

### 阶段 D：主工程适配
- 目标：主工程通过接口适配调用外部模块。
- 注意：执行器（DB、FS）继续留在主工程。

---

## 10. 测试策略

至少覆盖以下测试：

1. 规则匹配
- 优先级是否正确。
- 多关键词匹配是否符合预期。
- 未命中时是否返回 `Matched=false`。

2. overlay
- overlay 是否优先于基础规则书。
- overlay 过期后是否回退。

3. 集成
- 命中规则后目标目录是否正确。
- DB 字段是否按动作更新。
- EXDEV fallback 是否成功。

4. 回归
- 使用当前硬编码规则迁移后，旧行为不变。

---

## 11. 最小可用里程碑（MVP）

MVP 建议只做以下能力：
- RuleBook + Rule + Engine 基础实现
- 默认规则书迁移（与现有行为一致）
- `OSRelocationStep` 接入引擎
- 仓库级选择规则书（可先硬编码映射）

延后能力（第二期）：
- UI 配置规则书
- 规则书编辑审计日志
- 复杂条件表达式（正则、AND/OR 组合）

---

## 12. 决策结论

这个抽离方案不是“理论可行”，而是“工程上可落地且迁移成本可控”：
- 短期可提升可维护性（规则与业务解耦）。
- 中期可按仓库差异化策略。
- 长期可稳定迁出独立 Go module，供其他项目复用。

推荐立即启动阶段 A，并保持“先兼容后增强”的迁移原则。

---

## 13. 存储方案定稿（JSON + DB）

推荐采用混合模式：

1. JSON 文件：存放基线规则书（可版本化）
- 优点：易于 code review、可跟随代码发布、回滚简单。
- 场景：`default-os-relocation@v1`、`legacy-compatible@v1`。

2. DB：存放运行态绑定与临时覆盖规则
- 优点：支持按仓库差异化、运行时生效、审计与过期机制。
- 场景：repo 绑定哪本规则书、临时热修规则、灰度试验。

不建议“纯 DB”或“纯 JSON”：
- 纯 DB：基线规则难审阅，发布与回滚成本高。
- 纯 JSON：运行时动态能力不足，不利于临时修复与审计。

## 15. DB 表结构建议（运行态）

建议新增两张表：

### 15.1 repo_rulebook_bindings

用途：每个仓库当前绑定哪本规则书。

字段建议：
- `id` INTEGER PRIMARY KEY
- `repo_id` INTEGER NOT NULL UNIQUE
- `book_name` TEXT NOT NULL
- `book_version` TEXT NOT NULL
- `enabled` BOOLEAN NOT NULL DEFAULT true
- `updated_by` TEXT NOT NULL DEFAULT 'system'
- `created_at` DATETIME
- `updated_at` DATETIME

索引建议：
- UNIQUE(`repo_id`)
- INDEX(`book_name`, `book_version`)

说明：
- 一个 repo 只绑定一条当前生效记录。
- 若无绑定，则回退默认规则书（JSON）。

### 15.2 rulebook_overlays

用途：repo 级临时覆盖规则（热修、灰度）。

字段建议：
- `id` INTEGER PRIMARY KEY
- `repo_id` INTEGER NOT NULL
- `overlay_name` TEXT NOT NULL
- `rules_json` TEXT NOT NULL
- `priority_offset` INTEGER NOT NULL DEFAULT -1000
- `enabled` BOOLEAN NOT NULL DEFAULT true
- `expires_at` DATETIME NULL
- `updated_by` TEXT NOT NULL DEFAULT 'system'
- `reason` TEXT NOT NULL DEFAULT ''
- `created_at` DATETIME
- `updated_at` DATETIME

索引建议：
- INDEX(`repo_id`, `enabled`)
- INDEX(`repo_id`, `expires_at`)

说明：
- `rules_json` 存放一组 Rule（建议数组）。
- `priority_offset` 用于让 overlay 规则整体前置。
- `expires_at` 到期后自动失效，不需要删除。

---

## 16. 运行时加载优先级

每次构建 repo 规则快照，按以下顺序：

1. 加载 JSON 默认规则书（base）
2. 查询 repo 绑定（若有，切换到指定 base）
3. 查询 overlay（`enabled=true` 且 `expires_at` 未过期）
4. 将 overlay 规则应用 `priority_offset` 后 merge
5. 按 `priority` + 声明顺序排序
6. 产出不可变快照并交给 engine

建议日志字段：
- `repo_id`
- `book_name`
- `book_version`
- `overlay_count`
- `snapshot_hash`

## 17. 管理 API 草案

建议新增 API：

1. `GET /repos/:id/rulebook`
- 返回当前生效规则书元信息与 overlay 摘要。

2. `PUT /repos/:id/rulebook/binding`
- 请求体：`book_name`, `book_version`
- 效果：切换 repo 绑定规则书。

3. `POST /repos/:id/rulebook/overlays`
- 请求体：`overlay_name`, `rules`, `expires_at`, `reason`
- 效果：创建临时覆盖规则。

4. `PATCH /repos/:id/rulebook/overlays/:overlayId`
- 修改启用状态、过期时间。

5. `POST /repos/:id/rulebook/reload`
- 手动触发缓存刷新（可选）。

建议所有写接口记录：
- 操作人
- 变更前后摘要
- 变更原因

## 18. 实施顺序（建议）

1. 先做 JSON loader + 默认文件
- 目标：将默认规则从代码迁到文件，但行为不变。

2. 加 repo_rulebook_bindings
- 目标：支持每仓库切换规则书。

3. 加 rulebook_overlays
- 目标：支持临时规则与到期回退。

4. 补齐 API + 审计日志
- 目标：形成完整可运营能力。

5. 再考虑迁出外部 Go module
- 目标：将纯规则引擎层独立发布。

---

## 19. 已实现能力（截至当前）

当前仓库已实现 repo 级规则书绑定（最小可用）：

1. 绑定存储位置
- 存在 `repo.db` 的 `repo_info.flags_json` 字段中。
- 字段键名：`rulebook_name`、`rulebook_version`。

2. 生效方式
- `OSRelocationStep` 在执行时按 repo 读取绑定。
- 绑定文件路径规则：`backend/normalization/rules/{name}.{version}.json`。
- 若读取或构建失败，自动回退到内置规则书并打告警日志。

3. 绑定管理接口（已实现）
- `GET /api/repos/:id/rulebook/binding`
- `PUT /api/repos/:id/rulebook/binding`
- `GET /api/rulebooks`（列出可发现规则文件及校验状态）

4. 请求示例

```json
{
    "rulebook_name": "default-os-relocation",
    "rulebook_version": "v1"
}
```

5. 输入校验
- `rulebook_name` 正则：`^[a-z0-9][a-z0-9-]*$`
- `rulebook_version` 正则：`^v[0-9]+$`

6. 缓存行为
- 引擎按 `name@version` 缓存。
- 更新绑定后会清空缓存，下一次归类立即按新绑定加载。

7. 绑定前预检
- `PUT /api/repos/:id/rulebook/binding` 会先尝试加载目标规则文件。
- 若文件不存在或 JSON 无效，会直接返回 400，不会落库。

8. 默认与类型策略（已实现）
- 全局默认规则书：`noop@v1`。
- 无类型仓库（`repo_type=none`）预设绑定：`noop@v1`。
- OS 类型仓库（`repo_type=os`）预设绑定：`default-os-relocation@v1`。
- 当 OS 规则文件损坏时，容灾回退到 `noop`，表现为“停止自动移动”，并伴随状态告警。
